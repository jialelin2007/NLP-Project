# AGENTS.md

## Project Goal

This repository is for building a locally deployable English-to-Chinese translation model for CS/AI academic papers. The current target is full-parameter supervised fine-tuning of `Qwen/Qwen3-32B` on an 8 × RTX Pro 6000 96GB machine.

The final system should translate computer science and AI paper text into accurate, fluent, formal academic Chinese, with strong terminology consistency and good handling of long, complex sentences.

This is an engineering project, not only a proposal. Prefer changes that make the pipeline runnable, reproducible, and easy to evaluate.

## Current Experiment Direction

Use `Qwen/Qwen3-32B` as the main backbone. This project use multi-GPU full-parameter fine-tuning, because the available hardware is 8 × RTX Pro 6000 96GB.

Default training mode:

- Full-parameter SFT, not LoRA/QLoRA, unless explicitly asked to add a parameter-efficient baseline.
- Use all 8 GPUs by default.
- Use BF16 when supported.
- Use DeepSpeed ZeRO-3 or PyTorch FSDP for sharded training. Do not use naive DDP for full-parameter 32B training.
- Enable gradient checkpointing / activation checkpointing by default for memory safety.
- Prefer non-thinking translation behavior. Do not train the model to output hidden reasoning or `<think>...</think>` content.

Acceptable implementation choices include Transformers + TRL + DeepSpeed, Transformers + Accelerate/FSDP, LLaMA-Factory, Axolotl, or another mature fine-tuning framework if it fits the repository better. Do not rewrite a working framework unless necessary.

## Data Sources

The project should use a mixture of general translation data, scientific bilingual data, and CS/AI paper-style data.

Main intended data components:

1. General parallel translation data, such as `quickmt zh-en`, for basic translation ability.
2. Scientific bilingual data, such as CSL abstracts, filtered toward engineering, computer science, and AI.
3. CS/AI paper pseudo-parallel data constructed from arXiv and ACL Anthology papers. Typical English fields include abstracts, introductions, methods, and selected technical sections. Chinese targets may be generated by a strong teacher model and then filtered.

Use these data sources as project-level guidance. If exact dataset names, locations, or formats differ in the repository, inspect the existing files and adapt rather than forcing a new structure.

## Data Format

Prefer a clean intermediate JSONL or Parquet format before conversion to model-specific chat format.

Recommended intermediate schema:

```json
{
  "id": "unique_example_id",
  "source": "English source text",
  "target": "Chinese translation",
  "domain": "general|scientific|cs_ai_paper",
  "split": "train|validation|test",
  "metadata": {
    "source_dataset": "quickmt|csl|arxiv|acl|other",
    "paper_id": null,
    "section": null
  }
}
```

Recommended SFT chat format:

```json
{
  "messages": [
    {
      "role": "system",
      "content": "You are a professional academic translator. Translate English CS/AI paper text into accurate, fluent, formal Chinese. Preserve technical terms, equations, citations, code, variable names, and LaTeX syntax. Do not add explanations."
    },
    {
      "role": "user",
      "content": "Translate the following English academic text into Chinese:\n\n<source text>"
    },
    {
      "role": "assistant",
      "content": "<target Chinese translation>"
    }
  ]
}
```

When training, compute loss only on the assistant translation tokens if the framework supports label masking.

## Data Quality Rules

Implement practical filters, but do not over-engineer them before a runnable baseline exists.

Useful filters:

- Remove empty, duplicated, or extremely short examples.
- Filter obvious language mismatches.
- Filter severe length-ratio outliers between English and Chinese.
- Preserve equations, citations, references, LaTeX commands, code snippets, and variable names.
- Remove examples whose target includes model self-talk, refusals, unrelated commentary, or hidden reasoning tags.
- Keep a small validation and test set that are not mixed into training.

Maintain a small AI/CS terminology glossary, for example `data/glossary/ai_cs_terms.csv`, with columns such as `en`, `zh`, `aliases`, and `notes`. The glossary can be used for data checking, prompt-based glossary injection, post-processing, and terminology evaluation.

## Training Plan

Use a two-stage SFT pipeline unless the user asks for a different design.

Stage 1: General and Scientific Translation Adaptation

- Train on general translation data plus scientific bilingual data.
- Goal: stable English-to-Chinese translation behavior and formal scientific style.

Stage 2: CS/AI Paper Specialization

- Continue training from Stage 1 on CS/AI paper data.
- Goal: better AI terminology, paper-style sentence structure, and domain-specific phrasing.

Also support a single-stage training option for ablation if simple to implement.

Recommended initial hyperparameter ranges for full-parameter SFT:

- `max_seq_len`: start with 4096; increase to 8192 only after memory is stable.
- `learning_rate`: around `5e-6` to `2e-5`.
- `lr_scheduler`: cosine or linear decay.
- `warmup_ratio`: around `0.03`.
- `weight_decay`: around `0.01` to `0.1`.
- `gradient_clipping`: around `1.0`.
- `global_batch_size`: choose based on memory; use gradient accumulation.
- `packing`: optional; use only if it does not break loss masking or evaluation clarity.

Before launching a long run, run a short preflight training job by temporarily
overriding the formal training config to a small step count. Verify that the
job runs without NaN/OOM and that logging/checkpoint paths are correct.

## Evaluation

Evaluate at least the following systems when possible:

1. Base `Qwen/Qwen3-32B` without fine-tuning.
2. Stage 1 checkpoint.
3. Stage 2 checkpoint.
4. Optional ablation: full fine-tuning without CS/AI domain data.
5. Optional external baseline: a general translation model such as NLLB-200-distilled-600M.

Core metrics:

- SacreBLEU for lexical translation quality.
- COMET or another semantic metric for meaning preservation.
- Terminology accuracy: whether important AI/CS terms match the glossary.
- Terminology consistency: whether the same term is translated consistently across examples.
- Small human evaluation set for accuracy, fluency, academic style, and terminology correctness.

Evaluation scripts should save both metrics and sample translations. Prefer outputs like:

```text
outputs/eval/<run_name>/metrics.json
outputs/eval/<run_name>/samples.jsonl
outputs/eval/<run_name>/terminology_report.csv
```

## Translation Behavior Requirements

The model should learn to:

- Translate into Simplified Chinese by default.
- Use formal academic writing style.
- Preserve equations, LaTeX commands, citations, URLs, code, variable names, and section numbers.
- Translate technical terms consistently.
- Avoid adding explanations, summaries, opinions, or extra content.
- Avoid thinking traces, hidden reasoning, and `<think>` tags.
- Keep paragraph structure close to the source unless readability clearly improves.

For inference, prefer non-thinking mode and direct translation prompts.

## Running Long Jobs

For long 8-GPU jobs, prefer scripts that are restartable and log clearly.

Recommended behavior:

- Save training configs together with each run.
- Save tokenizer and model config with checkpoints.
- Log Git commit hash when possible.
- Save stdout/stderr logs under `outputs/logs/`.
- Support resuming from the latest checkpoint.
- Avoid deleting checkpoints unless the user explicitly asks.

If OOM occurs, try these adjustments in order:

1. Lower per-device batch size.
2. Increase gradient accumulation.
3. Lower max sequence length.
4. Ensure activation checkpointing is enabled.
5. Ensure ZeRO-3/FSDP sharding is actually active.
6. Consider CPU/NVMe offload only if necessary, because it may slow training significantly.

## What Codex Should Prioritize

First priority: make the project runnable end to end on a tiny sample.

A good first milestone is:

1. Environment can be created.
2. A tiny dataset can be prepared.
3. Qwen3 tokenizer can load.
4. The formal SFT training entry point can run for a few preflight steps.
5. Evaluation script can generate translations and compute at least SacreBLEU.
6. README documents the exact commands.

After that, scale up data, sequence length, and training duration.

When a requested task is complete, run the relevant verification commands first,
then commit the related git files before reporting completion unless the user
explicitly asks not to commit.
